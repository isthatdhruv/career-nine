  <name>Task 1: Create CacheWarmingConfig to populate Redis caches on startup</name>
  <files>spring-social/src/main/java/com/kccitm/api/config/CacheWarmingConfig.java</files>
  <action>
Create a new @Configuration class that listens for ApplicationReadyEvent and warms critical caches by calling the @Cacheable methods.

**Implementation details:**

1. Create `CacheWarmingConfig.java` in `com.kccitm.api.config` package.
2. Annotate with `@Configuration`.
3. Implement `ApplicationListener<ApplicationReadyEvent>` (not @PostConstruct — ApplicationReadyEvent fires after all beans are ready AND after the server is listening, which means the app is fully functional before warming starts).
4. Inject `AssessmentQuestionController` and `MeasuredQualityTypesController` via `@Autowired`. Calling the controller methods directly works because Spring AOP proxies intercept the @Cacheable methods — the result gets cached in Redis automatically.

**Note on why inject controllers not repositories:** The @Cacheable annotation is on the controller methods. Calling the repository directly would bypass the cache and the data would not be stored in Redis. We must call the @Cacheable-annotated method through the Spring proxy to trigger cache population.

5. In `onApplicationEvent(ApplicationReadyEvent event)`:
   a. Wrap everything in a try-catch that logs WARN on failure (cache warming is best-effort — app must start even if Redis is down).
   b. Log `"Starting cache warming..."` at INFO level.
   c. Call `assessmentQuestionController.getAllAssessmentQuestions()` — this triggers @Cacheable("assessmentQuestions") and stores the full question list in Redis.
   d. Call `measuredQualityTypesController.getAllMeasuredQualityTypes()` — this triggers @Cacheable("measuredQualityTypes") and stores all MQT types in Redis.
   e. Do NOT warm "assessmentDetails" — this cache is keyed by individual assessment ID, so there's no single call to warm all entries. Individual entries get cached on first access.
   f. Log `"Cache warming completed successfully"` at INFO level.

**Why not warm assessmentDetails:** The cache key is `#id` (per-assessment). Warming would require iterating all assessment IDs and calling getAssessmentDetailsById for each. With many assessments, this is expensive at startup. Better to let these warm on-demand — the first student request for each assessment warms it.

6. Add `private static final Logger logger = LoggerFactory.getLogger(CacheWarmingConfig.class);`

<task type="auto">
  <name>Task 2: Add @Cacheable to prefetch endpoint for Redis-backed fast responses</name>
  <files>spring-social/src/main/java/com/kccitm/api/controller/career9/AssessmentTableController.java</files>
  <action>
The current prefetch endpoint at `/assessments/prefetch/{userStudentId}` queries the database every time. Add caching so repeated prefetch calls for the same student serve from Redis.

**Implementation details:**

1. Add `@Cacheable(value = "assessmentDetails", key = "'prefetch-' + #userStudentId")` to the `prefetchAssessmentData` method. Use a prefixed key to avoid collision with the per-assessment-ID cache entries that use `key = "#id"`.

2. The existing `@CacheEvict(value = "assessmentDetails", allEntries = true)` on create/update/delete/lock/unlock endpoints will automatically invalidate prefetch cache entries too (since allEntries=true clears the entire "assessmentDetails" cache). No additional eviction configuration needed.

**Why use "assessmentDetails" cache:** The prefetch returns assessment config data (name, isActive, showTimer, isLocked, questionnaireType). This is the same domain as assessmentDetails. When an assessment is updated/locked/unlocked, the existing @CacheEvict annotations clear allEntries=true, which correctly invalidates both per-ID and per-student prefetch entries.

**Alternative considered:** A separate "prefetchData" cache name. Rejected because it would require adding @CacheEvict for the new cache name to every mutation endpoint — error-prone and redundant. Using the existing "assessmentDetails" cache with a prefixed key is simpler and correct.

3. No other changes to the prefetch method body.
